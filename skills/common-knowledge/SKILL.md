# 通用开发知识库

适用于所有项目类型、语言和框架的通用开发最佳实践。这些实践无论使用何种技术栈都能保证代码质量、可维护性、安全性和操作卓越性。

## 代码质量指南

在所有项目中建立和维护高质量的代码标准：

- **遵循语言规范**：遵守语言特定的习语和风格指南（Python 的 PEP 8、Go 的 Google Style 等）
- **自我文档化代码**：使用清晰、有意义的名称来命名变量、函数和类
- **单一职责**：保持函数/方法小且专注于一个任务
- **有意义的命名**：使用清楚反映意图的变量名
- **SOLID 原则**：应用设计原则来维护面向对象代码
- **代码审查标准**：建立清晰的同行审查期望
- **风格一致性**：使用 linter 和格式化工具维护一致的代码风格

**使用场景**：
- 编写新代码或重构现有代码
- 建立团队编码标准
- 代码审查和质量评估
- 设置开发环境

[code_quality.md](./code_quality.md) - 完整的代码质量指南

## 依赖管理指南

有效且安全地管理外部依赖：

- **最小化依赖**：保持外部依赖数量到最少必需
- **定期更新**：保持所有依赖的版本最新
- **锁定文件**：使用版本锁定文件确保可重现构建
- **安全扫描**：定期扫描已知漏洞
- **许可证合规**：添加依赖前验证许可证兼容性
- **弃用追踪**：监控已弃用的依赖

**使用场景**：
- 添加新库或框架
- 日常维护和更新
- 安全审计和评估
- 构建可重现性设置

## 安全指南

在开发的所有层实现安全最佳实践：

- **密钥管理**：永远不要硬编码密钥、API 密钥或凭据
- **输入验证**：清理和验证所有外部输入
- **SQL 注入防护**：使用参数化查询和 ORM
- **身份认证/授权**：实现适当的访问控制机制
- **HTTPS 通信**：确保所有外部 API 使用加密通信
- **依赖安全**：保持依赖更新以获取安全补丁
- **OWASP Top 10**：应对常见 Web 应用漏洞
- **代码审查**：在同行审查中包含安全检查

**OWASP Top 10 类别**：
1. 注入攻击（SQL、命令等）
2. 破损的身份认证
3. 敏感数据泄露
4. XML 外部实体（XXE）
5. 破损的访问控制
6. 安全配置错误
7. 跨站脚本（XSS）
8. 不安全的反序列化
9. 使用含有已知漏洞的组件
10. 日志和监控不足

**使用场景**：
- 安全代码审查
- 威胁建模和风险评估
- 漏洞修复
- 合规性要求

## 测试指南

建立全面的测试策略以提高可靠性和质量：

- **测试金字塔**：70% 单元测试、20% 集成测试、10% E2E 测试
- **单元测试**：使用 Mock 依赖在隔离状态下测试业务逻辑
- **集成测试**：测试组件交互和集成点
- **E2E 测试**：从用户角度验证完整工作流
- **测试独立性**：每个测试必须独立且可重复
- **覆盖率目标**：目标覆盖率 >80%
- **测试命名**：使用描述性名称说明测试场景
- **Mock 策略**：为隔离测试而 Mock 外部依赖

**测试框架**：
- **Go**：testify、gocheck
- **Python**：pytest、unittest
- **Node.js**：Jest、Mocha
- **Java**：JUnit、TestNG

**使用场景**：
- 开发期间（推荐使用 TDD 方法）
- 代码审查和 PR 批准前
- 回归测试以防止 Bug
- 性能和负载测试

## 文档指南

创建清晰、全面、易于维护的文档：

- **README 文件**：项目概览、快速开始、安装说明
- **API 文档**：端点、参数、响应、错误码
- **架构文档**：系统设计、数据流、部署架构
- **代码注释**：解释复杂逻辑和非显而易见的实现选择
- **故障排除**：常见问题和解决方案
- **变更日志**：跨版本追踪变更
- **示例**：常见用例的工作代码示例

**文档格式**：
- **Markdown**：README、指南、通用文档
- **OpenAPI/Swagger**：API 规范和交互式文档
- **图表**：架构可视化、数据流、系统设计
- **代码注释**：复杂逻辑的内联说明

**使用场景**：
- 项目入门和设置
- API 使用和集成
- 故障排除和调试
- 架构审查和决策
- 发布说明和版本管理

## 版本控制指南

建立有效的版本控制实践：

- **提交消息**：清晰、描述性的信息解释"为什么"而不仅是"什么"
- **传统提交**：使用 `feat:`、`fix:`、`docs:`、`refactor:` 等格式
- **特性分支**：在合并前在独立分支上开发特性
- **主分支**：保持主分支始终可部署
- **代码审查**：合并前需要同行审查
- **PR 描述**：记录变更和测试方法
- **问题链接**：在提交和 PR 中引用问题号
- **发布标签**：使用语义版本标记发布

**语义版本化**：`MAJOR.MINOR.PATCH`
- **MAJOR**：破坏性变更
- **MINOR**：新特性（向后兼容）
- **PATCH**：Bug 修复（向后兼容）

**使用场景**：
- 日常开发工作流
- 代码审查和协作
- 发布管理
- 审计和合规追踪

## 错误处理指南

在整个应用程序中实现稳健的错误处理：

- **区分错误**：区分预期错误和意外错误
- **有意义的消息**：提供清晰的错误描述
- **错误上下文**：保留堆栈跟踪和上下文信息
- **优雅恢复**：适当地从可恢复错误中恢复
- **错误日志**：用完整上下文记录所有错误以便调试
- **错误类型**：在可用时使用类型化错误
- **不要忽略**：永远不要静默忽略错误
- **重试逻辑**：在适当的地方实现重试机制

**错误处理模式**：
1. 尽早验证输入 → 返回特定错误
2. 处理预期失败 → 使用错误类型
3. 记录意外错误 → 包含完整上下文
4. 向调用者返回错误 → 不要隐藏问题
5. 实现重试 → 适当的地方

**使用场景**：
- 编写业务逻辑
- API 设计和实现
- 调试生产问题
- 系统可靠性和监控

## 性能指南

在保持代码清晰的同时优化性能：

- **先进行基准测试**：测量后再优化 - 识别真正的瓶颈
- **关注瓶颈**：先优化最慢的组件
- **测量影响**：验证优化确实改进了性能
- **可读性 vs 性能**：平衡性能与代码可维护性
- **避免过早优化**：不要优化不需要的代码

**优化领域**：
- **数据库**：查询优化、索引、连接池
- **缓存**：内存缓存、分布式缓存、CDN
- **并发**：并行化、线程池、异步操作
- **算法**：时间/空间复杂性分析和改进
- **I/O 操作**：批处理、异步操作、压缩

**监控性能**：
- 追踪响应时间和延迟
- 监控资源使用（CPU、内存）
- 性能降低时告警
- 定期进行性能测试

**使用场景**：
- 性能测试和基准测试
- 生产监控和优化
- 扩展决策
- 架构审查

## 指南之间的整合

这些指南共同工作以确保整体项目质量：

1. **代码质量** + **测试** = 可靠的代码
2. **安全** + **依赖管理** = 安全的依赖
3. **文档** + **错误处理** = 易维护的系统
4. **版本控制** + **性能** = 可追踪的改进
5. **测试** + **性能** = 经过验证的优化

## 关键原则

- **一致性**：在所有项目中一致地应用实践
- **自动化**：尽可能自动化检查和验证
- **简洁性**：选择可行的最简单方案
- **演进**：基于经验不断改进实践
- **学习**：记录决策并从错误中学习

## 参考资源

- **SOLID 原则**：可维护代码的设计模式
- **DRY（不重复）**：避免代码重复
- **KISS（保持简单）**：偏好简洁胜过复杂
- **YAGNI（你不会需要它）**：不要构建不必要的特性
